REFACTOR: Start/Stop Architecture — Full Implementation Guide
=============================================================

This document is a complete implementation guide for Codex or any developer
implementing the start/stop architecture refactor. Read it entirely before
touching any code. Every decision here has been reasoned through carefully.


BACKGROUND AND MOTIVATION
--------------------------

The current codebase models the VPN interface lifecycle as:
  createInterface -> applyMtu -> applyAddresses -> applyRoutes -> applyDns -> packetIO -> deleteInterface

This worked for Linux where TUN devices are persistent OS objects that can be
created, configured, and deleted independently of whether any process is reading
from them. However, this model is fundamentally broken for Windows and macOS:

On Windows, TUN devices are created by the Wintun DLL when you open a handle.
The device exists only while the handle is open. When the handle closes, the
device is destroyed. There is no way to create a Wintun device and then apply
netsh configuration before opening the packet IO handle, because the device
does not exist until the handle is opened.

On macOS, utun interfaces are created by opening a kernel control socket
(SYSPROTO_CONTROL). The interface exists only while the socket is open. When
the socket closes, the interface is destroyed. The same problem applies.

The current codebase works around this by making createInterface() a no-op on
Windows and macOS (empty ExecutionPlan), which means any applyMtu, applyAddresses,
applyRoutes, or applyDns calls made before packetIO() will fail or be silently
lost because the interface does not exist yet. This is broken and unacceptable.

The solution is to make all three platforms behave like Windows and macOS:
interface lifetime equals session lifetime. On Linux this means switching from
"ip tuntap add" (persistent) to tun-rs (ephemeral, destroyed on handle close).
This loses the ability to pre-configure an interface before bringing it up, but
gains consistent behavior across all platforms.

The new model is:
  startSession(config, outgoingPackets) -> Flow<ByteArray>

One call. The daemon atomically creates the TUN device, applies all configuration
(mtu, addresses, routes, dns), and opens the packet bridge. When the caller stops
collecting the returned flow, the session ends and the TUN device is destroyed.


CORE DESIGN DECISIONS
----------------------

Decision 1: Single RPC replaces eight RPCs
The eight separate RPCs (createInterface, applyMtu, applyAddresses, applyRoutes,
applyDns, interfaceExists, readInterfaceInformation, deleteInterface) all become
one: startSession(config, outgoingPackets). The configuration is bundled into a
TunSessionConfig object sent with the call. This eliminates the ordering problem
entirely because the TUN is created and configured atomically inside one daemon
operation. The caller no longer has to think about which calls must happen before
or after which other calls.

Decision 2: TunSessionConfig carries pre-computed routes, not peers
The daemon receives a TunSessionConfig with a routes field (List<String>). Routes
are pre-computed by the caller from VpnConfiguration.peers.flatMap { allowedIps }
in an extension function in the core lib. The daemon must not receive VpnPeer
objects because peers carry WireGuard private and public keys that the daemon
(a privileged process) has no business seeing. Keep crypto concerns in the core
lib and network configuration concerns in the daemon.

Decision 3: DnsConfig replaces Pair<List<String>, List<String>>
The current dnsDomainPool field is a Pair where .first is search domains and
.second is DNS server addresses. Pair is opaque. Any caller can swap the two
lists and the compiler will not catch it. This creates silent misconfiguration
that is hard to debug. Replace with DnsConfig(searchDomains: List<String>,
servers: List<String>) which has named fields. This change applies everywhere
dnsDomainPool is used: VpnConfiguration, TunSessionConfig, VpnInterfaceInformation.

Decision 4: reconfigure() means stop then start, with a brief packet gap
Because TUN lifetime equals session lifetime, changing configuration requires
destroying the current TUN and creating a new one with the new config. This means
there will be a brief packet drop during reconfigure. This is an intentional
behavioral regression accepted in exchange for cross-platform consistency. The
rollback-on-partial-failure behavior that existed before (if applyDns failed,
roll back to old config) no longer exists. If startSession fails, the state is
Stopped and the caller must retry.

Decision 5: Client-side stability window is kept
The current openPacketBridge method in DaemonBackedInterfaceCommandExecutor
waits 200ms after the bridge starts to detect startup failures before reporting
success to the caller. This distinction matters because a flow that fails after
200ms is a steady-state runtime failure (call onFailure), while a flow that
fails immediately is a startup failure (throw synchronously). This logic is
preserved in the new openSession() method. The protocol does not add a
separate ack RPC for this purpose.

Decision 6: VpnState becomes Stopped and Running only
The current VpnState.Created state means "interface exists but is not running."
With ephemeral TUNs this state is impossible. The interface only exists while
a session is active. VpnState.NotCreated and VpnState.Created are removed.
VpnState.Stopped replaces VpnState.NotCreated. VpnState.Running is kept.

Decision 7: ProcessLauncher and shell utilities are kept
The new PlatformAdapter implementations still need to run shell commands for
routes and DNS after the TUN is created. ProcessLauncher, CommonsExecProcessLauncher,
CommandBinary, ProcessInvocationModel, and ProcessOutputModel survive as internal
utilities used directly by the new adapter implementations. Only the intermediate
abstraction layer (DaemonOperation, ExecutionPlan, PlanExecutor,
PlatformOperationPlanner) is removed.

MODULE 2: wg-kotlin-daemon-jvm
--------------------------------

This module is the privileged daemon process. It implements DaemonProcessApi.
This module has the most changes because the internal architecture changes
completely.

INTERNAL ARCHITECTURE OVERVIEW

The current architecture:
  DaemonProcessApiImpl
    -> PlatformOperationPlanner (translates DaemonOperation to ExecutionPlan)
    -> PlanExecutor (runs each ExecutionStep via ProcessLauncher)
    -> TunHandleFactory (separate concern, opened lazily in packetIO)

The problem: PlatformOperationPlanner can only produce shell commands.
Wintun DLL calls and utun socket creation are not shell commands so they were
modeled as empty ExecutionPlans, pushing the real work into DaemonProcessApiImpl
with type-checks like "if (handle is RealTunHandle) handle.openDevice()".

The new architecture:
  DaemonProcessApiImpl
    -> PlatformAdapter (encapsulates everything: TUN creation + config application)

PlatformAdapter.startSession(config) does all of:
1. Creates the TUN device via tun-rs (handles all platforms)
2. Applies routes using shell commands (after TUN exists)
3. Applies DNS using shell commands (after TUN exists)
4. Returns an open TunHandle ready for packet IO

DaemonProcessApiImpl just calls adapter.startSession(config), loops on
handle.readPacket(), writes outgoingPackets to handle.writePacket(), and on
flow completion calls handle.close() which destroys the TUN device.

FILE: src/main/kotlin/.../planner/PlatformAdapter.kt (CREATE)

  internal interface PlatformAdapter {
      val platformId: String
      suspend fun startSession(config: TunSessionConfig): TunHandle
  }

  internal object PlatformAdapterFactory {
      fun fromOs(osName: String = System.getProperty("os.name")): PlatformAdapter {
          val normalized = osName.lowercase()
          return when {
              arrayOf("mac", "darwin").any { normalized.contains(it) } -> MacOsPlatformAdapter()
              normalized.contains("win") -> WindowsPlatformAdapter()
              normalized.contains("linux") -> LinuxPlatformAdapter()
              else -> throw IllegalStateException("Unsupported OS: $osName")
          }
      }
  }

Why: The interface captures the only thing DaemonProcessApiImpl needs to know:
given a config, give me an open TunHandle. How the TUN is created (Wintun DLL,
utun socket, or tun-rs on Linux) is completely hidden from the caller.

FILE: src/main/kotlin/.../planner/LinuxPlatformAdapter.kt (CREATE)

This adapter:
1. Calls tun-rs (via RealTunHandleFactory or TunDevice directly) to create
   the TUN device. On Linux, tun-rs opens /dev/net/tun and creates a named
   TUN interface. When the file descriptor closes the interface is destroyed.
   This replaces the old "ip tuntap add" approach to get consistent ephemeral
   behavior with Windows and macOS.
2. After TUN is open, applies MTU via: ip link set dev <name> mtu <mtu>
3. Applies addresses via: ip address flush dev <name>, then ip address add <addr> dev <name> for each address
4. Applies routes via: ip route replace <route> dev <name> for each route
5. Applies DNS via resolvectl. If domains and servers are non-empty:
   resolvectl dns <name> <servers...>
   resolvectl domain <name> <~domain...>
   If empty: resolvectl revert <name>
6. Returns the open TunHandle

Use ProcessLauncher to run each shell command. Throw CommandFailed if any
command returns a non-zero exit code that is not in the accepted set.

Why: Linux previously used "ip tuntap add" which created a persistent interface.
Switching to tun-rs makes Linux behave like Windows and macOS: interface is
destroyed when the handle closes. This is the whole point of the refactor.

FILE: src/main/kotlin/.../planner/MacOsPlatformAdapter.kt (CREATE)

This adapter:
1. Calls tun-rs to create a utun interface. The utun interface is created by
   opening a kernel control socket. The interface name is assigned by the OS
   (utun0, utun1, etc.) unless tun-rs supports requesting a specific name.
   The interface is destroyed when the socket is closed.
2. After TUN is open, applies MTU via: ifconfig <name> mtu <mtu>
3. Applies addresses via: ifconfig <name> inet <addr> alias for IPv4,
   ifconfig <name> inet6 <addr> add for IPv6
4. Applies routes via: route -n delete -net <route> -interface <name> (ignore failure),
   then route -n add -net <route> -interface <name>
5. Applies DNS via scutil with stdin commands. See MacOsOperationPlanner for the
   exact scutil script format (the logic is the same, just move it here).
6. Returns the open TunHandle

Why: The old MacOsOperationPlanner returned an empty plan for CreateInterface
and added a comment saying "handled elsewhere." The actual utun socket creation
was inside RealTunHandle.openDevice() called via a type-check in DaemonProcessApiImpl.
This adapter collapses that into one place.

FILE: src/main/kotlin/.../planner/WindowsPlatformAdapter.kt (CREATE)

This adapter:
1. Calls WindowsDllLoader.loadWinTun() to ensure the Wintun DLL is loaded.
2. Calls tun-rs to create a Wintun adapter with the given interface name.
   The adapter is created by the Wintun DLL and exists only while the handle
   is open.
3. After TUN is open, applies MTU via netsh interface ipv4 set subinterface
4. Applies addresses via netsh interface ipv4/ipv6 add/delete address
5. Applies routes via netsh interface ipv4/ipv6 add route
6. Applies DNS via PowerShell NRPT rules. See WindowsOperationPlanner for the
   exact PowerShell script format (move the logic here verbatim).
7. Returns the open TunHandle

Important: on Windows, cleaning up DNS NRPT rules must happen when the session
ends, not when it starts. Add cleanup logic inside TunHandle.close() or as a
teardown step in DaemonProcessApiImpl's finally block after the session ends.

Why: WindowsOperationPlanner returned empty plans for CreateInterface and
DeleteInterface and could not model Wintun DLL calls at all. The adapter
owns the complete lifecycle.

FILE: DaemonProcessApiImpl.kt (REWRITE)

The new implementation:

  class DaemonProcessApiImpl(
      private val adapter: PlatformAdapter = PlatformAdapterFactory.fromOs(),
  ) : DaemonProcessApi {

      private val activeSessionLock = Any()
      private val activeSessions = mutableSetOf<String>()

      override suspend fun ping(): CommandResult<PingResponse> {
          return CommandResult.success(PingResponse)
      }

      override fun startSession(
          config: TunSessionConfig,
          outgoingPackets: Flow<ByteArray>,
      ): Flow<ByteArray> = channelFlow {
          DaemonPayloadValidator.validate(config)

          synchronized(activeSessionLock) {
              if (activeSessions.contains(config.interfaceName)) {
                  throw IllegalStateException("Session already active for ${config.interfaceName}")
              }
              activeSessions.add(config.interfaceName)
          }

          val handle = try {
              adapter.startSession(config)
          } catch (failure: Throwable) {
              synchronized(activeSessionLock) { activeSessions.remove(config.interfaceName) }
              throw failure
          }

          val writerJob = launch {
              outgoingPackets.collect { packet ->
                  if (packet.size > MAX_PACKET_FRAME_SIZE) {
                      throw IllegalArgumentException("Oversized packet: ${packet.size}")
                  }
                  if (packet.isNotEmpty()) handle.writePacket(packet)
              }
          }

          try {
              while (isActive) {
                  val packet = handle.readPacket() ?: continue
                  if (packet.size > MAX_PACKET_FRAME_SIZE) {
                      throw IllegalStateException("Oversized packet from TUN: ${packet.size}")
                  }
                  if (packet.isNotEmpty()) send(packet)
              }
          } finally {
              runCatching { writerJob.cancelAndJoin() }
              runCatching { handle.close() }
              synchronized(activeSessionLock) { activeSessions.remove(config.interfaceName) }
          }
      }

      private fun <S> toFailureResult(commandType: String, failure: Throwable): CommandResult<S> {
          // Keep existing toFailureResult logic, same exception types
      }
  }

Notice: the SetInterfaceState (up/down) calls are gone. The TUN is considered
"up" from the moment startSession returns successfully. There is no separate
up/down step. The TUN is destroyed when the flow ends.

Why: The old implementation called planExecutor.run(SetInterfaceState(up=true))
before starting packet IO and SetInterfaceState(up=false) in the finally block.
With tun-rs managing the TUN lifecycle, the interface state is tied to the handle
state. No explicit up/down commands are needed.

FILE: src/jvmMain/.../iface/TunSessionConfigFactory.kt (CREATE)

  internal fun VpnConfiguration.toTunSessionConfig(): TunSessionConfig {
      val routes = peers
          .flatMap { peer -> peer.allowedIps }
          .filter { route -> route.isNotBlank() }
          .distinct()
          .sorted()
      return TunSessionConfig(
          interfaceName = interfaceName,
          mtu = mtu,
          addresses = addresses.toList(),
          routes = routes,
          dns = DnsConfig(
              searchDomains = dns.searchDomains,
              servers = dns.servers,
          ),
      )
  }

Why: The daemon should not receive VpnPeer objects because peers carry WireGuard

not key material. Pre-computing routes here keeps crypto concerns in the core lib
and network configuration concerns in the daemon. This factory function is in
jvmMain because TunSessionConfig is from the daemon-protocol module which is
JVM-only.

FILE: src/jvmMain/.../iface/JvmInterfaceManager.kt (REWRITE)

The new implementation:

  class JvmInterfaceManager(
      private val commandExecutor: InterfaceCommandExecutor,
      private val tunPipe: DuplexChannelPipe<ByteArray>,
  ) : InterfaceManager {

      private var currentConfig: VpnConfiguration? = null
      private var activeBridge: AutoCloseable? = null

      override fun isRunning(): Boolean = activeBridge != null

      override fun start(config: VpnConfiguration, onFailure: (Throwable) -> Unit) {
          stop()
          activeBridge = commandExecutor.openSession(
              config = config.toTunSessionConfig(),
              pipe = tunPipe,
              onFailure = { throwable ->
                  activeBridge = null
                  currentConfig = null
                  onFailure(throwable)
              },
          )
          currentConfig = config
      }

      override fun stop() {
          runCatching { activeBridge?.close() }
          activeBridge = null
          currentConfig = null
      }

      override fun reconfigure(config: VpnConfiguration) {
          stop()
          start(config)
      }

      override fun information(): VpnInterfaceInformation? {
          val config = currentConfig ?: return null
          return VpnInterfaceInformation(
              interfaceName = config.interfaceName,
              isUp = isRunning(),
              addresses = config.addresses.toList(),
              dns = config.dns,
              mtu = config.mtu,
              listenPort = config.listenPort,
          )
      }
  }

Notice that the constructor no longer takes interfaceName as a separate parameter
because the name comes from the VpnConfiguration passed to start(). The manager
is no longer tied to a specific interface name at creation time, which is correct
because the interface only exists during a session.

Also notice that information() reads from the cached config, not from the daemon.
The daemon no longer exposes readInterfaceInformation. This means information()
returns null when no session is active and returns config-derived info when a
session is active.

Why: The old JvmInterfaceManager tracked interfaceName, currentConfiguration,
up boolean, and activeBridge as separate pieces of state. In the new model these
all collapse: the session IS the interface, so activeSession (the bridge handle)
is the only state that matters.

Remove NotCreated and Created. Search the entire codebase for all references
to VpnState.NotCreated and VpnState.Created and replace with VpnState.Stopped.

Why: NotCreated and Created were two different "not running" states distinguished
by whether the interface existed on the OS. With ephemeral interfaces this
distinction is meaningless. From a caller's perspective "stopped" is "stopped"
regardless of whether a kernel object happens to exist.
